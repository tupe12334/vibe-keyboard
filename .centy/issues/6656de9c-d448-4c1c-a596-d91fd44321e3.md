# Replace all unwrap() calls to eliminate crash risks

## Problem

The codebase uses `.unwrap()` calls (and potentially `.expect()`) in various places, which can cause the application to panic at runtime if the underlying `Option` or `Result` is `None`/`Err`. This is a reliability risk — any unhandled case will crash the entire app.

## Goal

Audit and replace every instance of `.unwrap()` and bare `.expect()` with proper error handling:

- Use `?` propagation where the caller can handle errors
- Use `if let` / `match` for intentional branching
- Use `unwrap_or`, `unwrap_or_else`, or `unwrap_or_default` where a fallback makes sense
- Replace panicking `.expect()` messages with recoverable error paths

## Acceptance Criteria

- [ ] `grep -rn '.unwrap()' src/` returns no results (or only in test code)
- [ ] `cargo clippy` passes with `#![deny(clippy::unwrap_used)]` or equivalent lint
- [ ] No new panics introduced; all error paths are handled gracefully
